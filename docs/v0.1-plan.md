# V0.1 Plan - MVP Telegram Flight Deal Bot

## Status

Shipped.

- Release tag: `v0.1.0`
- Scope: MVP for Telegram-based domestic Vietnam flight search, comparison, alerts, and booking/search links.
- Current production has moved past V0.1 into V0.2 hotfixes, but this document records the V0.1 baseline and the acceptance bar for future regressions.

## Product Boundary

AirDeal VN Bot stays inside this workflow:

```text
Search -> Compare -> Alert -> Send booking link
```

V0.1 explicitly does not include:

- Booking automation.
- Payment handling.
- Airline account login on behalf of users.
- PostgreSQL, Redis, Celery, or horizontal worker infrastructure.
- LLM parsing by default.
- Web app or Zalo bot.

## Objective

Deliver a small, production-deployable Telegram bot that can:

- Parse common Vietnamese flight search messages.
- Search domestic Vietnam flights through real or safe fallback providers.
- Compare offers by normalized `FlightOffer`.
- Let users create and manage route alerts.
- Send clear booking or search links without overstating what the link does.
- Run from Docker Compose with SQLite WAL and a health endpoint.

## Scope

### Bot Commands

- `/start`: welcome user and upsert Telegram user.
- `/help`: show supported commands and examples.
- `/search` and natural-language messages: parse route/date/passengers and return top offers.
- `/watch`: create or update price alert.
- `/alerts`: list active alerts.
- `/pause`: pause an alert for a duration.
- `/delete`: soft-delete an alert.
- `/deals`: show recent great deals when enough snapshot data exists.
- `/history`: show route price history when enough snapshot data exists.

### NLP

- Regex-first parser.
- Vietnamese airport aliases through `AIRPORT_ALIASES`.
- Relative dates through `dates_vi.py`.
- Round-trip parsing when input has both departure and return dates, including same-day return when explicitly labeled as `di/ve` or `đi/về`.
- Passenger parsing for adults, children, and infants.
- No LLM call unless a later version explicitly enables `USE_LLM_PARSER`.

### Flights

- Provider contract: every provider returns `list[FlightOffer]`.
- Fanout across enabled providers.
- Merge by `flight_key`.
- Keep cheapest offer per `flight_key`.
- Cache search results in SQLite for 30 minutes.
- Use safe outbound URLs only.

### Alerts

- Store alerts in SQLite.
- APScheduler scans active alerts on a configurable interval.
- Trigger when price is below max price or meaningfully drops.
- Deduplicate notifications by `flight_key`.
- Enforce daily alert quota per user.

### Operations

- Dockerfile and Docker Compose deployment.
- SQLite WAL mode.
- FastAPI `/health` endpoint.
- Telegram webhook secret verification.
- Per-user token bucket rate limit.
- Structured JSON logging with `structlog`.
- Config loaded through `app/config.py`.

## Key Decisions

- `flight_key = sha1(airline|flight_no|date|depart_time)`.
- SQLite WAL is sufficient for MVP and remains the storage engine until scale triggers V0.5 migration.
- Regex parser is the default NLP path; LLM parsing is a later optional fallback.
- Provider implementations must normalize into `FlightOffer`; no provider-specific response format leaks into bot handlers.
- Links shown to users must be honest:
  - Direct booking URLs can be labeled as booking links.
  - Search/result-page URLs must be labeled as search links.

## Providers

V0.1 supports a provider chain with safe degradation:

- Mock provider for deterministic tests.
- FastFlights provider as a no-key fallback for Google Flights search context.
- Atadi web provider through Playwright/CloakBrowser when enabled.
- Atadi REST provider only when `ATADI_API_KEY` is configured.

Provider failures must not crash the bot. If all providers fail, the bot replies with a user-facing retry message and logs provider errors.

## Success Criteria

V0.1 is acceptable when:

- A user can search a common domestic VN route from Telegram.
- A user can create, list, pause, and delete alerts.
- Search results show airline, flight number when present, times, price per person, total price, and a safe link.
- Parser rejects invalid date, route, and passenger inputs with Vietnamese user-facing messages.
- Rate limiting protects message handlers.
- Webhook mode verifies Telegram secret token.
- `/health` returns database and provider status.
- Tests pass locally.
- Docker Compose starts the bot and health endpoint.
- No secrets are committed or logged.

## Verification Baseline

The original V0.1 scaffold reached:

- 31 tests passing at the scaffold completion checkpoint.
- Git initialized and first commits split by project roadmap.
- Production deployment later continued into V0.2 work.

Current repository baseline has moved beyond V0.1:

Use V0.1 success criteria as regression guard, not as the current full feature list.

## Out of Scope For V0.1

- Price prediction beyond storing history and basic baseline context.
- Rich conversation wizard.
- Flexible month heat-map UI.
- Currency conversion.
- Official paid provider integrations that need new commercial approval.
- PostgreSQL, Redis, Celery, Sentry, Prometheus.
- Automated browser login to airline accounts.

## Risks And Mitigations

| Risk | Mitigation |
|---|---|
| Provider DOM changes break scraping | Keep providers isolated, log provider failures, keep fallback provider enabled |
| SQLite write contention | WAL mode, short transactions, migrate to PostgreSQL only when scale triggers are reached |
| Regex parser misses natural input | Reply with examples; add tested aliases/patterns incrementally |
| Alert spam | Dedup by `flight_key`, cooldown, daily quota |
| Misleading links | Validate URL and label search links separately from booking links |
| Secret leakage | Read secrets from env only; never commit `.env`; avoid logging tokens |

## Handoff To V0.2

V0.2 starts after V0.1 by improving usefulness, not expanding product scope:

- Better multi-provider comparison.
- Safer link labeling.
- Baseline-aware deal scoring.
- Snapshot recording from normal searches.
- Provider latency/error observability.
- Atadi CloakBrowser stabilization.
