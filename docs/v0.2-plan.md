# V0.2 Plan — Multi-provider + Real Deals

## Objective

Make AirDeal VN Bot useful beyond a working MVP by improving comparison quality and deal context while staying inside the product boundary:

```text
Search -> Compare -> Alert -> Send booking link
```

V0.2 does not add booking, payment, airline login, PostgreSQL, Redis, or LLM parsing.

## Scope

- Use multiple enabled providers in fanout and keep the cheapest offer per `flight_key`.
- Preserve real booking URLs from providers that return them.
- Provide a safe search URL for sources that only expose flight data, clearly labeled as a search link.
- Record search result snapshots so baseline percentiles can improve outside alert scans.
- Rank and display search results with baseline-aware deal badges when enough data exists.
- Keep SQLite WAL and the existing provider interface.

## Success Criteria

- FastFlights offers include a safe Google Flights search URL, not a direct booking claim.
- Search responses prefer score ranking when a route has enough baseline data and fall back to price otherwise.
- Search responses show P25/P15 deal context and median savings when available.
- User searches insert `price_snapshots` for returned offers.
- Existing tests and lint pass.

## Boundaries

- Always: validate outbound URLs before showing them to users.
- Always: keep provider outputs as `FlightOffer`.
- Ask first: adding paid/public APIs, new database engines, or new long-running infrastructure.
- Never: automate booking/payment or log secrets.
